# Архитектура Workspace AndreyT (2026-04-17)

## Проблема которую решаем

Нужна система где:
1. Один пользователь (AndreyT) работает над НЕСКОЛЬКИМИ проектами и инициативами
2. При старте сессии ясно понимаем: проект + роль + ветка работы
3. Память каждого проекта отделена от других
4. Потеря памяти исключена (резервное копирование, версионирование)
5. Удобное переключение между проектами

## Решение: Структура /home/hive/projects/

```
/home/hive/projects/
│
├── .claude-session-config         ⭐ Конфиг workspace (идентификация AndreyT)
├── .gitignore
├── .git/                          (git инициализирован)
│
├── transport-crm/                 # Основной проект (разработка)
│   ├── src/
│   ├── CLAUDE.md                  (инструкции: роль=developer)
│   ├── PROGRESS.md                (статус проекта)

│   │   └── [другие файлы]
```

## Файл конфигурации: .claude-session-config

```json
{
  "owner": "AndreyT",
  "email": "andrey@example.com",
  "projects_root": true,
  "description": "Projects workspace for AndreyT",
  "roles": {
    "default": "project-owner",
    "sub_projects": {
      "transport-crm": "developer",
      "consulting": "consultant",
      "research": "researcher"
    }
  }
}
```

**Назначение:** При старте сессии я читаю этот файл и определяю что ты AndreyT.

## Система памяти (защита от потери)

### В каждом проекте/консультации:

```
memory/
├── MEMORY.md              # Индекс: список всех файлов памяти с описанием
├── _SUPER_CONTEXT.md      # ⭐ КРИТИЧНЫЙ КОНТЕКСТ (резервная копия)
├── [другие файлы]         # Детальные анализы, решения
```

### _SUPER_CONTEXT.md — "чёрный ящик" памяти

- Содержит САМОЕ ВАЖНОЕ (не потеряется)
- Критичные факты, дедлайны, контакты, решения
- Читается в ПЕРВУЮ очередь при старте
- Версионируется в Git

### MEMORY.md — индекс с описаниями

- Список всех файлов памяти в table-формате
- Помогает понять что есть в памяти
- Содержит краткую справку (что делаем, статус)

### Git версионирование памяти

```bash
# В конце каждой сессии:
git add projects/{project}/memory/
git commit -m "docs: обновление памяти {project}"
```

## Алгоритм старта сессии

Когда ты (AndreyT) пишешь мне сообщение:

```
1. Я проверяю CWD и родительские директории
2. Ищу .claude-session-config
3. Если найден и owner="AndreyT":
   ├─ Загружаю конфиг
   ├─ Определяю текущий проект (по CWD)
   ├─ Определяю роль из config.roles.sub_projects
   ├─ Читаю PROGRESS.md проекта
   ├─ Читаю memory/_SUPER_CONTEXT.md
   ├─ Проверяю целостность памяти (MEMORY.md)
   └─ Приветствую тебя:
      👋 AndreyT, добро пожаловать!
      📁 Проект: {project}
      👤 Роль: {role}
      ✅ Статус памяти: целостна
      📍 Где остановились: [из PROGRESS.md]
```

## Идентификация AndreyT при старте

**ДА, я вижу что ты AndreyT, если:**
- CWD = /home/hive/projects/ или его подпроект
- Файл .claude-session-config присутствует и содержит owner="AndreyT"
- В моём первом сообщении я обращусь к тебе по имени: "👋 AndreyT, добро пожаловать!"

**ЕСЛИ я не обращусь к тебе по имени** → значит конфиг не найден или что-то не так.

## Преимущества архитектуры

✅ **Инкапсуляция:** Каждый проект имеет свой CLAUDE.md, PROGRESS.md, memory/
✅ **Защита памяти:** _SUPER_CONTEXT.md + Git версионирование
✅ **Идентификация:** При старте из projects/ я сразу знаю что это AndreyT
✅ **Гибкость:** Легко добавлять новые проекты/консультации
✅ **История:** Все решения версионируются в Git
✅ **Переключение:** Просто меняешь CWD → меняется контекст

## Когда это нужно было?

- **Проблема:** Консультация по ДПТ потеряется когда мы вернёмся к Transport CRM
- **Решение:** Отделяем в separate project + memory + git
- **Масштабируемость:** Когда появится 3-й, 4-й проект — всё по одной схеме

## Что нужно делать в следующих сессиях

1. При старте — читаю PROGRESS.md + _SUPER_CONTEXT.md
2. Если что-то забыл → ищу в MEMORY.md → читаю нужный файл
3. Когда нужна новая консультация/проект → создаю в /projects/
4. В конце сессии → коммитим память в Git
