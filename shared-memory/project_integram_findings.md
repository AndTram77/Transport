Integram — разрабатываемая платформа, CRM "Мой Парк" является тест-кейсом на ней.
Оценка сделана 2026-04-25 по итогам построения реального проекта.

**Why:** Андрей тестирует Integram как продукт — нужно зафиксировать честные выводы
**How to apply:** При обсуждении доработок платформы — опираться на эти наблюдения

## Сильные стороны
- Концепция lookup-таблиц + VALCOL — правильная архитектура
- MCP-сервер — конкурентное преимущество (мало у кого есть)
- REST API позволяет строить кастомный фронтенд поверх

## Критические баги (нужно исправить до показа заказчику)

### 1. Валюта-поле хранит ref по совпадению ID
Если числовое значение валюта-поля совпадает с существующим ID объекта в workspace,
Integram сохраняет ссылку вместо числа. Данные портятся без предупреждения.
Пример: Комиссия парка = 720₽ → сохраняется как ref на объект id:720.
Обходного пути через API нет.

### 2. confirm_action сломан (404 / постоянная ошибка)
Без confirm_action невозможно удалять записи через MCP.
Мусорные записи при ошибочном вводе не удаляются — остаются навсегда.

### 3. Тип колонки нельзя изменить после создания
Создана как ref → навсегда ref. Нет ALTER COLUMN, нет миграций схемы.
Рейтинг у 6 водителей сломан — нельзя починить без потери данных.

### 4. VALCOL не документирован
При создании записи через MCP нужно посылать `"ФИО": "..."` вместо `"value": "..."`.
Это нигде явно не написано — узнаётся только методом тыка.

### 5. MCP vs REST API — разные форматы ответа
MCP возвращает `"Водитель": "Иванов (id:345)"`, REST отдаёт `{value: "Иванов"}`.
Разработчик вынужден разбираться самостоятельно.

## Паттерны работы с Integram (для следующих сессий)

### Создание записи с VALCOL
```js
// НЕПРАВИЛЬНО:
{ "value": "Иванов Алексей" }  // → запись называется "Запись N"

// ПРАВИЛЬНО (VALCOL=ФИО):
{ "ФИО": "Иванов Алексей" }    // → запись называется "Иванов Алексей"
```

### Lookup tables (справочники)
Записи без кастомных колонок — только `name`. Создаются через create_object с пустым fields `{}`,
имя задаётся автоматически как `name` через системное поле.

### Числа vs ID коллизии
Избегать передавать целые числа в валюта/число поля если они могут совпасть с ID объектов.
Особенно опасен диапазон активных IDs в workspace (всё что < 1000 в малых workspace).

## Рекомендации для доработки платформы
1. Validation: валюта-поле должно отклонять значения совпадающие с существующими IDs
2. Починить confirm_action в MCP
3. Добавить ALTER COLUMN (смена типа с конвертацией данных)
4. Документировать VALCOL для MCP-разработчиков
5. Унифицировать формат ответа MCP и REST API
